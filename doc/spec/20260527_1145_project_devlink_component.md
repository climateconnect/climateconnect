## Problem Statement

Some projects need a custom-designed description section instead of the standard rich-text description. Currently the project detail page always renders the same `ProjectContent` component with the project's `description` field as formatted text. There is no way to replace this with a richer, Webflow-designed layout.

The hub system already solved a similar problem: `Hub.landing_page_component` stores a devlink component name, and the hub landing page renders that component instead of a default view. The same pattern should be available for project descriptions â€” a project can optionally specify a devlink component name, and when set, the project description tab renders that component instead of the normal description block.

The devlink component is **pure visual content** â€” no interactive callbacks, no dynamic data injection. It renders with its Webflow defaults (hardcoded text, images, links). This is the simplest possible integration and matches the current hub landing page pattern.

**Core Requirements:**

- Add a `devlink_component` field to the Project model that stores a devlink component name (e.g., `"Wasseraktionswochen"`).
- Create an auto-generated component registry (`devlink/componentRegistry.ts`) that maps component names to their statically imported references. This registry is generated by a script that runs after `npx webflow devlink sync` (integrated into the existing `fix-devlink.js` post-sync pipeline).
- In `ProjectContent` (the first tab's content), when `project.devlink_component` is set, render the corresponding devlink component **instead of** the normal description section. The metadata above the description (created date, organizer, collaboration info) and the progress/timeline section below remain unchanged.
- **Language support**: The devlink component name stored in the database uses a language prefix convention (e.g., `EnChErlangenLandingpage`). The prefix is `En` for English, `De` for German. At render time, the prefix is swapped to match the user's locale before looking up the component in the registry. For example, if the database stores `EnChErlangenLandingpage` and the user's locale is `de`, the system looks up `DeChErlangenLandingpage` in the registry. If the locale-matched component is not found, fall back to the original (stored) component name.
- When `devlink_component` is not set, the project detail page behaves identically to today.
- All project tabs (`#project`, `#team`, `#comments`, `#registrations`) continue to function normally. Only the description block within the first tab is affected.

**Explicitly Out of Scope:**

- Passing props or callbacks to the devlink component (the component renders with its Webflow defaults).
- Migration of existing Wasseraktionswochen references to use this mechanism.
- Hub landing page changes (the Hub model's `landing_page_component` field is unrelated to this task).
- Per-language component name storage (only one name is stored; the prefix swap handles language switching).

### Non-Functional Requirements

- **No breaking changes** to existing API contracts or data models. The new field is nullable and optional.
- **Backward compatible**: projects without `devlink_component` behave identically to today.
- **Registry auto-generation**: the component registry must be regenerated automatically as part of the devlink sync workflow (`npx webflow devlink sync` â†’ `node scripts/fix-devlink.js` â†’ registry generation). No manual step required.

## Error Handling and Corner Cases

The `getDevlinkComponent(name, locale)` function must handle the following cases gracefully. In all error cases, the normal project description block is rendered â€” no crash, no error boundary needed.

| Scenario | Behavior |
|----------|----------|
| `devlink_component` is `null` or empty string | No lookup attempted. Normal description rendered. |
| Component name set, exists in registry | Render the devlink component. |
| Component name set, NOT in registry (typo, or component removed after Webflow sync) | Log a console warning. Normal description rendered. |
| Prefixed name (e.g. `EnChErlangenLandingpage`), locale matches `en` | Look up `EnChErlangenLandingpage`. Render if found, else normal description. |
| Prefixed name, locale is `de`, swapped version `DeChErlangenLandingpage` exists | Render `DeChErlangenLandingpage`. |
| Prefixed name, locale is `de`, swapped version NOT found | Log a console warning. Normal description rendered. Do NOT fall back to the English component â€” showing a German component on the English site (or vice versa) looks like a bug. |
| No recognized prefix (e.g. `Wasseraktionswochen`), exists in registry | Render it. No locale switching. |
| No recognized prefix, NOT in registry | Log a console warning. Normal description rendered. |
| Admin enters a sub-component name (e.g. `Button`, `OrgCard`) | Component renders but will look wrong â€” this is a data integrity issue. The help text guides against it. No programmatic guard needed. |

Console warnings should use a consistent format: `[devlink] Component "X" not found in registry (locale: Y).` This helps developers debug missing component issues without breaking the page.

- [ ] The project API serializer (`ProjectSerializer`) includes `devlink_component` in its fields.
- [ ] `scripts/fix-devlink.js` (or a new script called from it) generates `devlink/componentRegistry.ts` by scanning all `.d.ts` files in `devlink/`.
- [ ] The generated registry maps each exported component name to its static import.
- [ ] A `getDevlinkComponent(name, locale)` utility function exists that: (a) detects the `En`/`De` prefix, (b) swaps the prefix based on locale, (c) looks up the registry, (d) returns `undefined` if not found â€” no fallback to the original name when a prefix was detected (mismatched language looks like a bug), (e) for names without a recognized prefix, looks up the name directly without locale switching.

- [ ] When `devlink_component` is not set, the project detail page renders identically to the current behavior (no regression).
