## Problem Statement

An event organiser or team admin can cancel the registration of an individual guest directly from the event's Registrations tab. This allows admins to react to unwanted or invalid registrations (e.g. bots, spam accounts, or mistaken sign-ups) without requiring manual database intervention.

**Core Requirements (User/Stakeholder Stated):**

- On the event's Registrations tab, each active guest row has a per-row action menu (e.g. a dot/three-dot menu) that includes a **"Cancel registration"** option.
- Clicking "Cancel registration" opens a **confirmation step** (modal) before the cancellation is processed.
- The confirmation step includes an **optional plain-text message** field. If filled in, the message body is emailed to the cancelled guest. No subject is entered by the admin — the subject is auto-generated by the system using the event name.
- After confirmed cancellation:
  - The `EventRegistration` record is **soft-deleted**: `cancelled_at` is set to now and `cancelled_by` is set to the requesting admin user (using the `cancelled_by` field introduced in [#1850](https://github.com/climateconnect/climateconnect/issues/1850)).
  - The freed seat is immediately available for other members to register.
  - If the event's `EventRegistrationConfig.status` was `FULL`, it reverts to `OPEN`.
  - The cancelled row **remains visible** in the Registrations tab with a clear **"Cancelled"** indicator. Admins need to see the full history of registrations including cancellations.
  - If the admin entered a message, it is emailed to the guest. If no message was entered, no email is sent.
- The `GET /api/projects/{slug}/registrations/` endpoint must now return **all registrations** (active and cancelled), so admins can see the full registration history after cancellations. Each row includes `id` (so the frontend can call the delete endpoint) and `cancelled_at` (so the frontend can derive the active/cancelled status).

**Explicitly Out of Scope (this iteration):**
- The guest cancelling their own registration — that is [#1850](https://github.com/climateconnect/climateconnect/issues/1850).
- The guest re-registering after an admin cancellation — the re-registration logic from [#1850](https://github.com/climateconnect/climateconnect/issues/1850) handles this generically; no additional work needed here.
- Rich-text or HTML message bodies in the optional notification email.
- Persisting the optional cancellation message to the database — the message is a one-time email only.
- An organiser-facing log of all cancellation messages sent.
- The UX details specific to [#1871](https://github.com/climateconnect/climateconnect/issues/1871): cancellation count summary at the bottom of the table (e.g. "100 guests (5 cancellations)"), tooltip on the status icon, and the option to toggle cancellations in/out of the print/export view. Those are handled in [#1871](https://github.com/climateconnect/climateconnect/issues/1871) which builds on this task.

### Non Functional Requirements

- The cancel endpoint must be **server-side permission-checked**: only users with organiser or team admin role may cancel a guest's registration (`403 Forbidden` otherwise). The UI hiding the action menu is not sufficient.
- The cancellation (soft delete + seat restoration) must be **atomic** — wrapped in a database transaction.
- An admin must not be able to cancel an already-cancelled registration. The endpoint returns `400 Bad Request` if the target registration already has `cancelled_at IS NOT NULL`.
- No breaking changes to existing API contracts. The `GET /api/projects/{slug}/registrations/` response gains new fields (`id`, `cancelled_at`) but existing fields are unchanged — additive only.

### AI Agent Insights and Additions

- **`cancelled_by` reuse from [#1850](https://github.com/climateconnect/climateconnect/issues/1850)**: the `cancelled_by` FK on `EventRegistration` was introduced in [#1850](https://github.com/climateconnect/climateconnect/issues/1850) and explicitly designed for this future admin-cancellation task. When an admin cancels, `cancelled_by` is set to the requesting admin user — no model or migration change is needed. This is the key distinction in the data layer between a guest self-cancellation and an admin cancellation.

- **Separate endpoint from member self-cancellation**: `DELETE /api/projects/{slug}/registrations/` is the member's own cancellation endpoint (no ID segment — user inferred from the auth token). Admin cancellation targets a *specific* registration, so the URL includes the registration ID: `DELETE /api/projects/{slug}/registrations/{registration_id}/`. Keep them as separate endpoints with separate permission checks — do not conflate the two.

- **Resolves the `TODO #1850` comment in the list view**: the `ListEventRegistrationsView` introduced in [#1863](https://github.com/climateconnect/climateconnect/issues/1863) contains a `TODO #1850: add .filter(cancelled_at__isnull=True) once cancelled_at is added` comment. This task resolves that TODO by doing the *opposite*: removing any such filter so all registrations (active + cancelled) are returned to the admin. The serializer (`EventRegistrationSerializer`, formerly `EventParticipantSerializer`) must also expose `id` and `cancelled_at` so the frontend can work with them.

- **Overlap with [#1871](https://github.com/climateconnect/climateconnect/issues/1871)**: both this issue and [#1871](https://github.com/climateconnect/climateconnect/issues/1871) require the same backend change (list endpoint returning all registrations with `cancelled_at`). This task implements that shared backend change. [#1871](https://github.com/climateconnect/climateconnect/issues/1871) then builds on top to add the count summary, tooltip, and export improvements. **#1872 should be implemented first; #1871 immediately after.** Flag with the team whether [#1871](https://github.com/climateconnect/climateconnect/issues/1871) should be updated to mark its backend work as delivered by this task.

- **Optional message email — synchronous, single recipient**: unlike the bulk-send feature in [#1866](https://github.com/climateconnect/climateconnect/issues/1866), this is a single email to a single guest. A synchronous send (inline in the view, no Celery task) is appropriate — same pattern as the "Send test" path in [#1866](https://github.com/climateconnect/climateconnect/issues/1866). This keeps the implementation simple without sacrificing responsiveness. Email is skipped entirely when no message is provided.

- **Permission pattern**: follow the same inline `ProjectMember` queryset check used in `ListEventRegistrationsView` and `EditRegistrationConfigView` — no new permission class is needed.

- **DataGrid row actions — MUI community edition**: the community edition of `@mui/x-data-grid` (already installed from [#1863](https://github.com/climateconnect/climateconnect/issues/1863)) does not have a built-in row-actions column. Use a `renderCell` on a dedicated non-sortable, non-filterable, non-exportable "Actions" column with a three-dot `IconButton` and `Menu`. Only active rows (`cancelled_at === null`) show the "Cancel registration" menu item; cancelled rows do not show the action.

- **After cancellation — optimistic local update**: after a successful `204` response, update the matching row in local DataGrid state by setting `cancelled_at` to the current timestamp — no full refetch needed. Also notify the parent component to update the `available_seats` display in the project header (e.g. via an `onRegistrationChange` callback, consistent with the pattern used in the member cancel flow in [#1850](https://github.com/climateconnect/climateconnect/issues/1850)).

- **New Mailjet templates**: follow the existing `_TEMPLATE_ID` / `_TEMPLATE_ID_DE` naming convention and settings pattern established in [#1866](https://github.com/climateconnect/climateconnect/issues/1866). Two new templates are needed (EN + DE). The subject is auto-generated in code (not a Mailjet template variable) — localised to the recipient's language, e.g. "Your registration for [Event Name] has been cancelled".

## System Impact

- **Actors involved**:
  - `Organiser / Team Admin`: Cancels a specific guest's registration from the event's Registrations tab, with an optional explanatory message.
  - `System`: Soft-deletes the `EventRegistration` record, restores the available seat count, and — if a message was provided — emails the guest.
- **Actions to implement**:
  - `Admin` → `Cancel Guest Registration` → `EventRegistration` (soft delete: set `cancelled_at`, `cancelled_by`) + `EventRegistrationConfig` (available seat count restored)
  - `System` → `Notify Cancelled Guest` (conditional on message being provided) → email sent to the guest with auto-generated subject and the admin's message body
- **Flows affected**:
  - **New flow — Admin Cancel Guest Registration**: Admin views Registrations tab → opens per-row action menu → selects "Cancel registration" → confirmation modal with optional message → confirms → system soft-deletes `EventRegistration` → row shows "Cancelled" state in DataGrid.
  - **View Registration List Flow** ([#1863](https://github.com/climateconnect/climateconnect/issues/1863)): extended — now returns all registrations (active + cancelled); response shape gains `id` and `cancelled_at` fields. This is the same change that [#1871](https://github.com/climateconnect/climateconnect/issues/1871) also requires, so this task delivers that shared change.
- **Entity changes needed**: None — `EventRegistration` already has `cancelled_at` and `cancelled_by` from [#1850](https://github.com/climateconnect/climateconnect/issues/1850). No new fields or migrations.
- **Flow changes needed**: Yes — new admin cancellation flow; existing registration list flow extended.
- **Integration changes needed**: Yes — two new Mailjet email templates (EN + DE) for the guest cancellation notification.
- **New specifications required**: None — this spec is sufficient. After implementation, confirm with the team whether the backend changes here close the backend scope of [#1871](https://github.com/climateconnect/climateconnect/issues/1871).

## Software Architecture

### API

**Admin cancel a guest registration (new endpoint)**

```
DELETE /api/projects/{slug}/registrations/{registration_id}/
```

- Auth required — returns `401 Unauthorized` if unauthenticated.
- Requires organiser or team admin role (`role_type` in `["all", "read write"]`) — returns `403 Forbidden` otherwise.
- Returns `404 Not Found` if the project does not exist, registration is not enabled for the project, or no `EventRegistration` with `registration_id` exists on this project.
- Returns `400 Bad Request` if the targeted registration is already cancelled (`cancelled_at IS NOT NULL`).
- Sets `cancelled_at = now()` and `cancelled_by = request.user`, atomically.
- If `EventRegistrationConfig.status` was `FULL` and active registrations are now below `max_participants`, reverts status to `OPEN` — same logic as in [#1850](https://github.com/climateconnect/climateconnect/issues/1850).
- Accepts an **optional** `message` field in the request body. If provided and non-empty, sends a cancellation notification email to the guest (synchronous — single email, consistent with "Send test" in [#1866](https://github.com/climateconnect/climateconnect/issues/1866)). If absent or empty, no email is sent.
- Returns `204 No Content` on success.

| Status | Condition |
|--------|-----------|
| `204 No Content` | Cancellation successful |
| `400 Bad Request` | Registration already cancelled |
| `401 Unauthorized` | Unauthenticated request |
| `403 Forbidden` | Authenticated but not an organiser or team admin |
| `404 Not Found` | Project not found, registration not enabled, or `registration_id` not found on this project |

**Updated registration list endpoint (existing)**

```
GET /api/projects/{slug}/registrations/
```

- Now returns **all** registrations for the event — active and cancelled — by removing the `cancelled_at__isnull=True` filter (resolves the `TODO #1850` comment in `ListEventRegistrationsView`).
- Response adds two new fields per row (additive — no existing fields removed or changed):

| New field | Type | Description |
|-----------|------|-------------|
| `id` | integer | `EventRegistration` primary key — used by the frontend to build the delete URL |
| `cancelled_at` | ISO 8601 string or `null` | `null` = active registration; populated = cancelled |

- All existing fields (`user_first_name`, `user_last_name`, `user_url_slug`, `user_thumbnail_image`, `registered_at`) are unchanged.
- Existing 401/403/404 error responses are unchanged.

No other API changes.

### Events

No async Celery task is introduced by this story. The optional cancellation notification email is sent synchronously in the view (single recipient — latency is negligible). See the Backend section for the email helper details.

### Backend

- **`EventRegistrationSerializer`** (in `organization/serializers/event_registration.py`): add `id` and `cancelled_at` as read-only fields. These are exposed only in the organiser-facing list response — no impact on any other serializer or endpoint.

- **`ListEventRegistrationsView`** (in `organization/views/event_registration_views.py`): remove the `cancelled_at__isnull=True` queryset filter (or the `TODO #1850` placeholder comment). The view now returns all `EventRegistration` rows for the event — active and cancelled. No other changes to this view.

- **New view — `AdminCancelRegistrationView`** (in `organization/views/event_registration_views.py`):
  - Handles `DELETE /api/projects/{slug}/registrations/{registration_id}/`.
  - Wrap the full operation in `@transaction.atomic`.
  - Look up the project by `url_slug` — `404` if not found.
  - Inline organiser/admin permission check (same `ProjectMember` queryset pattern used in `ListEventRegistrationsView` and `EditRegistrationConfigView`) — `403` if not satisfied.
  - Look up the `EventRegistration` by `registration_id` scoped to this project's `EventRegistrationConfig` — `404` if not found on this project.
  - Return `400 Bad Request` if `cancelled_at IS NOT NULL` (already cancelled).
  - Set `cancelled_at = now()` and `cancelled_by = request.user`, then save.
  - If `EventRegistrationConfig.status == "FULL"` and active registrations drop below `max_participants`, revert `status` to `"OPEN"` (mirror the FULL→OPEN logic in the `destroy()` action from [#1850](https://github.com/climateconnect/climateconnect/issues/1850)).
  - If the request body contains a non-empty `message` field: call the new email helper synchronously to send the cancellation notification to the guest.
  - Return `204 No Content`.

- **New email helper** (in `organization/utility/email.py`): sends a guest cancellation notification. Accepts the guest `User`, the `Project`, and the admin's `message` body string. The helper:
  - Resolves the recipient's preferred language.
  - Auto-generates the email subject in the recipient's language (e.g. "Your registration for [Event Name] has been cancelled") — the subject is **not** a user-provided input.
  - Builds a localised event title and a language-aware event URL, following the same pattern as `send_event_registration_confirmation_to_user`.
  - Assembles the Mailjet template variables (see the Mailjet templates section below).
  - Delegates to the existing `send_email()` utility.

- **New Mailjet templates** (action item — not code): two new templates (EN + DE) for the guest cancellation notification email. Required template variables:

  | Variable | Description |
  |----------|-------------|
  | `FirstName` | Recipient's first name (falls back to username) |
  | `EventTitle` | Event name localised for the recipient |
  | `EventUrl` | Language-aware URL to the event page |
  | `OrganiserName` | Localised organisation name or organiser's display name (same logic as the registration confirmation email) |
  | `OrganizerMessage` | The plain-text message body provided by the admin |

  The email subject is set in code (not as a template variable) — it is auto-generated, not organiser-authored.

- **New settings** (in `climateconnect_main/settings.py`): add two settings reading from environment variables, placed next to the existing `EVENT_ORGANIZER_MESSAGE_TEMPLATE_ID` block:
  - `ADMIN_CANCEL_REGISTRATION_TEMPLATE_ID` — Mailjet template ID for the English template.
  - `ADMIN_CANCEL_REGISTRATION_TEMPLATE_ID_DE` — Mailjet template ID for the German template.
  Add both to `.backend_env` and to the Azure App Service configuration.

- **URL** (in `organization/urls.py`): register `DELETE projects/<str:url_slug>/registrations/<int:registration_id>/` pointing to `AdminCancelRegistrationView`. This must be placed **before** any catch-all URL patterns and is distinct from the existing `DELETE /registrations/` route (no ID segment) used for member self-cancellation.

- **Django migration**: none — `cancelled_at` and `cancelled_by_id` columns are already added by [#1850](https://github.com/climateconnect/climateconnect/issues/1850). No schema change is needed.

### Data

No schema changes. The `cancelled_by_id` column set to the admin user's ID is the data-level distinction between a guest self-cancellation and an admin cancellation. A future query (`WHERE cancelled_by_id != user_id`) can identify admin-initiated cancellations if needed.

### Other

- The optional cancellation message is **not persisted** to the database — it is emailed once and not stored. If a cancellation reason log is needed in the future, a `cancellation_message` field can be added to `EventRegistration` at that point.
- The Mailjet templates must be created and template IDs configured before the optional email can be sent in production. The backend does not error without them — `send_email` logs a warning if the template ID is blank — but no emails will be delivered.

## Technical Solution Overview

Backend implemented on 2026-04-09 alongside [`20260309_1500_member_cancel_event_registration.md`](./20260309_1500_member_cancel_event_registration.md) to share the soft-delete pattern and `FULL→OPEN` seat-revert logic.

### `DELETE /api/projects/{url_slug}/registrations/{registration_id}/` — `AdminCancelRegistrationView`
New class in `organization/views/event_registration_views.py`. Wrapped in `@transaction.atomic`. Flow:
1. Look up project by `url_slug` (with `select_related`/`prefetch_related` for email helper) → `404` if not found
2. Inline `ProjectMember` queryset check for `role_type in [ALL_TYPE, READ_WRITE_TYPE]` → `403` if not satisfied (same pattern as `EditRegistrationConfigView`)
3. Look up `EventRegistrationConfig` via `select_for_update()` → `404` if not found
4. Look up `EventRegistration` by `id=registration_id` scoped to the RC via `select_for_update()` → `404` if not found on this project
5. Guard: `cancelled_at IS NOT NULL` → `400 Bad Request`
6. Soft-delete: `cancelled_at = now()`, `cancelled_by = request.user`
7. Revert `EventRegistrationConfig.status` `FULL→OPEN` if active count drops below `max_participants`
8. Optional email: `message = request.data.get("message", "").strip()` — if non-empty, calls `send_guest_cancellation_notification()` synchronously. Email errors are caught and logged; cancellation is **never rolled back** due to email failure.
Returns `204 No Content`.
